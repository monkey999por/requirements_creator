# 企業ニュースモニタリング

## 概要

企業のエシカルスコアは静的なものではなく、日々のニュースや事件によって大きく変動する。データ漏洩、大量解雇、訴訟、環境問題などの重大イベントをユーザーが見逃すと、スコア変動の理由が理解できない。本機能は、ユーザーが利用中のサービスの運営企業に関する倫理的ニュースをAIが自動収集・分類し、影響度に応じてリアルタイムでアラート通知する。

## 画面構成

### ニュースフィード画面
- **タイムラインビュー**: 登録サービスの企業に関するニュースを時系列で表示。各ニュースにはインパクト度（高/中/低）のバッジと影響する軸のアイコン
- **フィルター**: 企業別、軸別（プライバシー/労働/環境/税務/安全性）、インパクト度別でフィルタリング
- **ニュースカード**: タイトル、AI要約（3行以内）、情報ソースリンク、スコアへの影響予測を表示

### ニュース詳細画面
- **記事要約**: AIが生成した詳細要約
- **スコア影響分析**: このニュースがどの軸のスコアにどの程度影響するかの予測表示
- **関連ニュースリスト**: 同企業の関連する過去ニュースへのリンク
- **アクションボタン**: 「代替を探す」「スコア詳細を見る」への導線

### 通知設定画面
- **アラートレベル設定**: 通知を受け取るインパクト度のしきい値を設定
- **通知チャンネル**: プッシュ通知、メール、アプリ内のみの選択
- **サイレント時間帯**: 通知を抑制する時間帯の設定

## ユーザーフロー

1. システムがバックグラウンドで登録サービスの企業に関するニュースを自動収集する
2. AIがニュースの内容を分析し、倫理的インパクト度と影響軸を判定する
3. インパクト度がユーザーの設定しきい値を超えた場合、プッシュ通知を送信する
4. ユーザーが通知をタップしてニュース詳細画面を開く
5. ユーザーがAI要約とスコア影響分析を確認する
6. ユーザーが必要に応じて「代替を探す」をタップし、代替レコメンド機能へ遷移する
7. ニュースがスコアリングバッチに反映され、ダッシュボードのスコアが更新される

## データモデル

### company_news テーブル
| カラム名 | 型 | 説明 |
|----------|------|------|
| id | UUID | 主キー |
| company_id | UUID | companies.id への外部キー |
| title | VARCHAR(500) | ニュースタイトル |
| source_url | TEXT | 元記事URL |
| source_name | VARCHAR(200) | メディア名 |
| published_at | TIMESTAMP | 記事公開日時 |
| fetched_at | TIMESTAMP | 取得日時 |

### news_analysis テーブル
| カラム名 | 型 | 説明 |
|----------|------|------|
| id | UUID | 主キー |
| news_id | UUID | company_news.id への外部キー |
| impact_level | ENUM('high', 'medium', 'low') | インパクト度 |
| affected_axes | TEXT[] | 影響する軸の配列 |
| summary | TEXT | AI生成要約 |
| score_impact_prediction | JSONB | 軸別スコア影響予測 |
| analyzed_at | TIMESTAMP | 分析日時 |

### user_notification_settings テーブル
| カラム名 | 型 | 説明 |
|----------|------|------|
| id | UUID | 主キー |
| user_id | UUID | users.id への外部キー |
| min_impact_level | ENUM('high', 'medium', 'low') | 通知対象の最低インパクト度 |
| channels | TEXT[] | 通知チャンネル（push/email/in_app） |
| silent_start | TIME | サイレント開始時刻 |
| silent_end | TIME | サイレント終了時刻 |

### user_notifications テーブル
| カラム名 | 型 | 説明 |
|----------|------|------|
| id | UUID | 主キー |
| user_id | UUID | users.id への外部キー |
| news_analysis_id | UUID | news_analysis.id への外部キー |
| is_read | BOOLEAN | 既読フラグ |
| sent_at | TIMESTAMP | 送信日時 |
| read_at | TIMESTAMP | 既読日時 |

## API設計

| メソッド | パス | 概要 |
|----------|------|------|
| GET | /api/news/feed | ユーザーの登録サービスに関するニュースフィードを取得 |
| GET | /api/news/:newsId | ニュース詳細（AI分析付き）を取得 |
| GET | /api/news/company/:companyId | 特定企業のニュース一覧を取得 |
| GET | /api/notifications/me | 未読通知一覧を取得 |
| PUT | /api/notifications/:notificationId/read | 通知を既読にする |
| GET | /api/notifications/settings | 通知設定を取得 |
| PUT | /api/notifications/settings | 通知設定を更新 |

## 非機能要件

- **データ収集頻度**: ニュースソースを1時間ごとにクロール。重大ニュース（キーワードマッチ）は15分間隔で検出
- **パフォーマンス**: ニュースフィードの表示は500ms以内。AI要約の生成は初回のみ10秒程度（以降はキャッシュ）
- **通知品質**: 誤検知（無関係なニュースのアラート）率を5%以下に抑える。通知の過多を防ぐため、同一企業への連続通知は最短4時間間隔
- **スケーラビリティ**: ニュース収集パイプラインはキューベースで設計し、企業数の増加に対して水平スケール可能
- **セキュリティ**: ニュース記事のキャッシュは著作権に配慮し、要約とメタデータのみ保存。元記事へのリンクで原典参照を促す
