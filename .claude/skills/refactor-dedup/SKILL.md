# コード重複削減リファクタリング

`$ARGUMENTS` を対象にコードの重複を調査し、共通化によるコード量削減を行う。

## 前提情報

- 現在のブランチ: !`git branch --show-current`
- リモート: !`git remote -v | head -1`

## 手順

### 1. 対象範囲の特定

`$ARGUMENTS` を解析し、対象のディレクトリ・ファイル群を特定する。
引数が曖昧な場合は `Glob` / `Grep` で候補を探し、ユーザーに確認する。

### 2. 重複パターンの調査

Exploreエージェントを使い、以下の重複パターンを網羅的に探索する:

| 優先度 | パターン | 削減効果 |
|--------|---------|---------|
| 高 | 同一SVG/アイコンの複数箇所でのインライン定義 | 大（1箇所あたり10-15行） |
| 高 | 同一UIパーツ（ボタン・ヘッダー等）の重複実装 | 大（1箇所あたり10-20行） |
| 高 | モバイル/デスクトップで同一コードの2重実装 | 大（数十〜数百行） |
| 中 | 同一ロジック（状態管理・API呼び出し）の重複 | 中（カスタムフック抽出） |
| 中 | 定数・設定値のハードコーディング重複 | 小〜中 |
| 低 | 類似だが微妙に異なるコード（props差分で吸収可能か判断） | 要検討 |

### 3. 調査結果の提示とユーザー確認

以下の形式で整理し、ユーザーに提示する:

```
## リファクタリング対象一覧

| # | パターン | 該当箇所 | 推定削減行数 | 優先度 |
|---|---------|---------|-------------|--------|
| 1 | <重複の説明> | <ファイル:行番号> × N箇所 | 約XX行 | 高/中/低 |
| 2 | ... | ... | ... | ... |

合計推定削減: 約XX行

上記の方針で進めてよいですか？除外・追加があれば指示してください。
```

**必ずユーザーの承認を得てから次のステップに進む。**

### 4. 共通コンポーネント・フックの設計

承認された各パターンに対して:

1. **共通化の方針を決定**
   - 新規共有ファイル作成 vs 同一ファイル内のローカル抽出
   - propsの設計（差分をpropsで吸収、デフォルト値の設定）
   - 既存の共有ディレクトリ構造に合わせた配置

2. **設計原則**
   - 最小限のpropsで最大の再利用性を確保
   - 既存の動作・スタイルを一切変更しない（見た目の同一性を保証）
   - オプショナルなカスタマイズはデフォルト値付きpropsで対応
   - motion/アニメーション付きの要素はラッパーで対応（コンポーネント自体はシンプルに）

### 5. 実装

以下の順序で実装する:

1. **共有ファイルの作成**（新規コンポーネント・フック）
2. **消費側ファイルの編集**（import追加 + インラインコードを共有コンポーネントに置換）
3. **不要になったimportの削除**

編集時の注意:
- 1ファイルずつ確実に編集する
- 置換対象のコードは前後のコンテキストを含めて一意に特定する
- `replace_all` は変数リネームなど全置換が明確な場合のみ使用

### 6. デグレードチェック

#### 6-1. 静的チェック

```bash
pnpm check
```

- Biomeのlint/formatエラーが出た場合は即座に修正
- `useExhaustiveDependencies` 等のフック依存配列エラーに特に注意
- 既存の（今回の変更と無関係な）エラーは無視してよい

#### 6-2. ビルドチェック

```bash
pnpm viewer:build
```

- TypeScriptコンパイルエラーがないことを確認
- import漏れ・型不一致を検出

#### 6-3. 目視確認ポイントの提示

ユーザーに以下を報告:

```
## デグレードチェック結果

### 自動チェック
- [x] pnpm check: パス / NN件の既知エラー（変更と無関係）
- [x] pnpm viewer:build: パス

### 手動確認推奨項目
- [ ] <確認項目1>（例: サイドバーのビューモードボタン動作）
- [ ] <確認項目2>（例: モバイル・デスクトップ両方の表示）
```

git commit -m "refactor: <変更内容の要約>

## 判断基準

### 共通化すべきケース

- 3箇所以上で同一コードが存在する
- 2箇所でも10行以上の同一コードがある
- SVG/アイコンのインライン定義が複数箇所にある

### 共通化すべきでないケース

- 1箇所でしか使われていないコード（将来の再利用を見越した早期抽象化は避ける）
- 見た目は似ているが意味的に異なるコード（Toast通知 vs メッセージパネル等）
- motion/アニメーションの `whileHover` 等が直接SVGに付いている場合（ラッパーで対応）
- 共通化のためのpropsが4つ以上必要になる場合（複雑化のサイン）

## 注意事項

- ユーザー確認なしでリファクタリングを開始しない
- 見た目・動作の変更は一切行わない（純粋なコード構造の改善のみ）
- force pushは絶対に行わない
- `--no-verify` でコミットフックをスキップしない
- リファクタリングと無関係な修正・改善を含めない
- 既存のテスト・ビルドが通ることを必ず確認してからコミットする
